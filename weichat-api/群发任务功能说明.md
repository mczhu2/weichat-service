# 群发任务功能说明

## 2. 功能目标

群发任务功能的目标是：

1. 支持从外部联系人或群聊中筛选目标。
2. 将一次群发请求拆分为一个任务主记录和多条发送明细。
3. 支持手动触发任务执行。
4. 支持定时任务自动扫描未发送明细并逐条发送。
5. 回写每条发送结果以及任务整体统计数据。

当前实现里，群发任务本质上是“任务编排 + 明细驱动 + 文本发送”。

## 3. 整体业务流程

### 3.1 主流程概览

完整流程如下：

1. 前端或调用方先通过目标查询接口圈选群发对象。
2. 调用创建任务接口，生成 `mass_task` 主表记录。
3. 系统按接收对象展开明细，生成多条 `mass_task_detail` 记录。
4. 后续通过两种方式执行发送：
   - 手动触发指定任务。
   - 定时任务自动扫描未发送明细。
5. 每次发送按明细逐条调用下游企业微信发送接口。
6. 发送完成后更新明细状态，并同步更新任务统计。

### 3.2 角色划分

- `MassMessageTargetController`
  负责目标查询，解决“发给谁”。
- `MassMessageController`
  负责任务创建、查询、触发、取消、删除等任务管理。
- `GroupMassMessageService`
  负责构造任务和明细，并支持手动触发整任务发送。
- `MassMessageService`
  负责单条明细的实际发送与结果回写。
- `MassMessageJob`
  负责定时扫描待发送明细并驱动发送。
- `MassTaskService`
  负责任务主表维护。
- `MassTaskDetailService`
  负责任务明细表维护。

## 4. 业务对象定义

### 4.1 群发任务 `MassTask`

表示一次群发行为的主记录，核心含义如下：

- `taskType`
  - `1`：给外部联系人群发
  - `2`：给群聊群发
- `msgType`
  - `0`：文本
  - `1`：图片
  - `2`：文件
  - `3`：音频
  - `4`：视频
  - `5`：小程序消息
- `sendStatus`
  - `0`：待发送
  - `1`：发送中
  - `2`：已完成
  - `3`：已取消

说明：

- 虽然数据模型已预留图片、文件、音频、视频素材字段，但当前发送逻辑实际只走文本发送链路。
- `msgType=5` 已作为小程序消息类型预留，但当前 `MassTask` 主表尚未补齐小程序消息所需的专属载荷字段，例如标题、描述、AppID、原始 ID、页面路径、封面或图标信息。
- `sendTime` 已入库，但当前调度逻辑未按该字段做时间控制。

### 4.2 群发任务明细 `MassTaskDetail`

表示任务拆分后的单个接收对象发送记录，核心含义如下：

- `receiverType`
  - `1`：外部联系人
  - `2`：群聊
- `isSent`
  - `0`：未发送
  - `1`：已发送过
- `sendStatus`
  - `0`：发送失败
  - `1`：发送成功

每一条 `MassTaskDetail` 对应一个实际接收对象，因此一次群发任务最终会落成多条明细。

### 4.3 消息模板 `MessageTemplate`

模板用于复用消息内容，当前主要支持：

- 保存模板内容
- 保存消息类型
- 发送前将模板变量替换为接收方名称

当前实际渲染逻辑只稳定使用 `{{name}}` 变量。

## 5. MySQL 模型与表关系

### 5.1 主表 `mass_task`

来源：`doc/sql/mass_message_tables.sql`

用途：

- 记录一次任务级别的群发行为
- 保存任务类型、内容、消息类型、模板 ID、统计结果、状态等

关键字段：

- `id`：任务主键
- `task_name`：任务名称
- `task_type`：任务类型，区分发给人还是发给群
- `content`：文本内容
- `image_media_id` / `file_media_id` / `audio_media_id` / `video_media_id`：素材 ID
- `msg_type`：消息类型
- `template_id`：模板 ID
- `send_status`：任务状态
- `total_count`：总接收对象数
- `sent_count`：已发送对象数
- `success_count`：发送成功对象数

### 5.2 明细表 `mass_task_detail`

用途：

- 记录任务下每个接收对象的独立发送状态
- 支撑重试、逐条发送、逐条统计和审计

关键字段：

- `id`：明细主键
- `task_id`：所属任务 ID
- `receiver_type`：接收方类型
- `receiver_id`：接收方主键 ID
- `receiver_name`：接收方名称快照
- `is_sent`：是否已发送
- `send_status`：发送是否成功
- `send_result`：失败原因或结果描述
- `send_time`：发送时间

### 5.3 模板表 `message_template`

用途：

- 保存复用消息模板
- 供任务创建和发送前渲染使用

关键字段：

- `template_name`
- `template_content`
- `variables`
- `msg_type`
- `creator`

### 5.4 表关系图

```text
message_template
    id
     ^
     |
     | template_id
     |
mass_task
    id
     ^
     |
     | task_id
     |
mass_task_detail
```

说明：

- `mass_task.template_id -> message_template.id`
- `mass_task_detail.task_id -> mass_task.id`
- `mass_task_detail` 对 `mass_task` 是典型一对多关系

### 5.5 业务数据依赖关系

群发任务本身还依赖业务基础表：

- `wx_user_info`
  用于外部联系人筛选、发送用户 ID 获取、接收方名称展示
- `wx_group_info`
  用于群聊筛选、群 `room_id` 获取、群名称展示

关联关系不是外键，而是业务逻辑上的读取关系：

- `MassTaskDetail.receiverId` 在 `receiverType=1` 时对应 `wx_user_info.id`
- `MassTaskDetail.receiverId` 在 `receiverType=2` 时对应 `wx_group_info.id`

## 6. 接口关系与职责

## 6.1 目标查询接口

### 6.1.1 查询外部联系人目标

- 路径：`GET /api/v1/mass-message-targets/external-contacts`
- 兼容路径：`GET /api/v1/query/external-contacts`

入参：

- `uuid`：可选，设备或账号维度过滤
- `corpIds`：可选，企业 ID 列表，支持多选
- `pageNum`
- `pageSize`

作用：

- 供前端圈人
- 返回符合条件的外部联系人列表

SQL 逻辑：

- 基于 `wx_user_info`
- `uuid` 非空时增加 `uuid = ?`
- `corpIds` 非空时增加 `corpid in (...)`
- `is_del = 0`

### 6.1.2 查询群聊目标

- 路径：`GET /api/v1/mass-message-targets/groups`
- 兼容路径：`GET /api/v1/query/groups`

入参：

- `corpIds`：必传，支持多选
- `uuid`：可选
- `pageNum`
- `pageSize`

作用：

- 供前端圈群
- 返回符合条件的群聊列表

SQL 逻辑：

- 基于 `wx_group_info`
- `corp_id in (...)`
- 若传 `uuid`，则通过 `wx_user_info` 反查群创建人归属设备

## 6.2 模板管理接口

路径前缀：`/api/v1/message-template`

包含能力：

- 创建模板
- 修改模板
- 删除模板
- 查询单个模板
- 分页查询模板列表

用途：

- 为群发任务提供模板内容来源

## 6.3 群发任务接口

路径前缀：`/api/v1/mass`

### 6.3.1 快捷创建接口

- `POST /send-to-users`
  给多个外部联系人创建文本群发任务

- `POST /send-to-users-with-template`
  给多个外部联系人创建模板群发任务

- `POST /send-to-groups`
  给多个群聊创建文本群发任务

- `POST /send-to-groups-with-template`
  给多个群聊创建模板群发任务

这几类接口的本质是：

1. 组装 `MassTask`
2. 根据目标 ID 列表生成 `MassTaskDetail`
3. 调用 `massTaskService.createMassTask(...)` 持久化

### 6.3.2 通用建任务接口

- `POST /task`

入参：

- `MassTask` 请求体
- `receiverIds`
- `receiverType`

作用：

- 支持按通用方式自行创建群发任务
- 由控制器现场展开任务明细

### 6.3.3 任务操作接口

- `POST /trigger-task/{taskId}`
  手动触发任务执行

- `GET /tasks`
  查询任务列表

- `GET /task/{taskId}`
  查询任务详情

- `GET /task-details/{taskId}`
  查询任务明细

- `PUT /cancel-task/{taskId}`
  取消任务

- `PUT /task/{taskId}`
  更新任务基础信息

- `DELETE /task/{taskId}`
  删除任务及其明细

- `GET /task-stats/{taskId}`
  查询任务统计信息

## 7. 创建任务的内部实现

### 7.1 创建文本任务

当调用以下接口时：

- `/send-to-users`
- `/send-to-groups`

内部逻辑是：

1. 构造 `MassTask`
2. 设置：
   - `taskType`
   - `content`
   - `msgType = 0`
   - `creator`
3. 调用 `buildUserDetails(...)` 或 `buildGroupDetails(...)`
4. 写入主表和明细表

### 7.2 创建模板任务

当调用以下接口时：

- `/send-to-users-with-template`
- `/send-to-groups-with-template`

内部逻辑是：

1. 先根据 `templateId` 查询 `message_template`
2. 若模板不存在则直接抛异常
3. 构造 `MassTask`
4. 将模板内容写入 `task.content`
5. 将模板消息类型写入 `task.msgType`
6. 设置 `task.templateId`
7. 按目标列表展开明细并入库

### 7.3 明细展开规则

对每一个接收方 ID：

- 生成一条 `MassTaskDetail`
- 设置 `receiverType`
- 设置 `receiverId`
- 根据目标表读取名称并写入 `receiverName`
- 初始状态：
  - `isSent = 0`
  - `sendStatus = 0`

这意味着：

- 一次任务的可发送总量等于接收对象数
- 后续发送与统计都依赖这些明细

## 8. 发送执行逻辑

## 8.1 手动触发

调用 `POST /api/v1/mass/trigger-task/{taskId}` 后：

1. 根据 `taskId` 查询任务主记录
2. 查询该任务下所有明细
3. 将任务状态更新为 `1`，即发送中
4. 依次遍历明细并调用 `sendSingleMassMessage(...)`
5. 统计成功数
6. 回写任务统计
7. 将任务状态更新为 `2`

### 8.1.1 手动触发的重要特征

- 逐条串行发送
- 当前代码不会主动跳过 `is_sent = 1` 的明细
- 因此重复触发同一任务存在重复发送风险

## 8.2 定时任务自动发送

定时任务类：`MassMessageJob`

执行方式：

- `@Scheduled(fixedRate = 60000)`
- 每 60 秒执行一次

执行逻辑：

1. 查询 `mass_task_detail` 中 `is_sent = 0` 的记录
2. 取最早创建的前 10 条
3. 对每条调用 `massMessageService.sendMassMessageToReceiver(...)`

说明：

- 定时任务是按“明细”维度发，不是按“任务”维度发
- 目前扫描条件只看明细是否已发，不看任务状态和发送时间

## 8.3 单条发送逻辑

`MassMessageService.sendMassMessageToReceiver(...)` 负责按接收方类型分流：

- `receiverType = 1`
  - 查询 `wx_user_info`
  - 获取 `user_id`
  - `isRoom = false`

- `receiverType = 2`
  - 查询 `wx_group_info`
  - 解析 `room_id`
  - `isRoom = true`

最终统一调用：

- 下游接口：`/wxwork/sendtext`

发送请求本质是：

```json
{
  "send_userid": 目标用户ID或群roomId,
  "isRoom": false 或 true,
  "content": "最终发送文本"
}
```

返回值判断规则：

- `code == 0`：发送成功
- 否则：发送失败，并记录 `msg`

## 9. 模板渲染逻辑

发送前若任务带有 `templateId`：

1. 根据 `templateId` 查询模板
2. 若模板仍存在，则以模板内容为准
3. 使用 `MessageTemplateUtil.renderTemplate(...)` 渲染

当前默认渲染变量：

- `{{name}}`

当前实现特点：

- 创建模板任务时虽然接口允许传 `variables`
- 但发送时并未使用调用方传入的动态变量 Map
- 当前稳定生效的是按接收者名称填充 `name`

## 10. 统计与状态流转

### 10.1 明细状态

明细发送成功时：

- `is_sent = 1`
- `send_status = 1`
- 写入 `send_time`

明细发送失败时：

- `is_sent = 1`
- `send_status = 0`
- 写入 `send_result`
- 写入 `send_time`

这里的语义是：

- “是否已发送”表示是否已经尝试发送过
- “发送状态”表示该次尝试是成功还是失败

### 10.2 任务状态

任务主表理论状态如下：

- `0`：待发送
- `1`：发送中
- `2`：已完成
- `3`：已取消

任务统计字段包括：

- `totalCount`
- `sentCount`
- `successCount`

更新方式：

- 明细成功或失败后，系统重新查询该任务下全部明细
- 根据 `isSent` 和 `sendStatus` 重算统计

## 12. 当前实现边界与已知问题

以下内容是当前代码实际行为，不是设计预期。

### 12.1 当前只支持文本实际发送

虽然任务表已包含：

- `image_media_id`
- `file_media_id`
- `audio_media_id`
- `video_media_id`
- `msg_type`

但当前发送链路统一调用的是 `/wxwork/sendtext`，没有根据 `msgType` 分流到图片、文件、音频、视频发送逻辑。

结论：

- 当前“群发任务”已具备多媒体字段建模
- 但尚未真正支持多媒体群发
- 小程序消息类型虽然已纳入 `msgType` 枚举，但还没有对应的小程序消息载荷建模和发送分流

### 12.2 `sendTime` 尚未参与调度

当前定时任务只筛选：

- `mass_task_detail.is_sent = 0`

没有判断：

- 任务是否已取消
- 任务是否到达 `sendTime`

结论：

- 现阶段不是严格意义上的定时发送
- 更接近“轮询所有未发送明细并立即执行”

### 12.3 取消任务后仍可能被定时任务继续发送

取消任务接口只是把：

- `mass_task.send_status = 3`

但定时任务取数时未关联 `mass_task` 状态。

结论：

- 若任务已有未发送明细，取消后仍可能被扫描并发送

### 12.4 手动触发存在重复发送风险

手动触发会取该任务下全部明细，没有过滤：

- `is_sent = 0`

结论：

- 同一任务重复触发时，已发送过的明细也可能再次发送

### 12.5 异常状态与状态定义存在语义偏差

在 `GroupMassMessageService.triggerMassTask(...)` 中，异常场景会把任务状态更新为 `3`。

但表定义里：

- `3` 实际表示已取消

结论：

- 当前异常失败和人工取消共用同一状态值，语义不够清晰

### 12.6 模板变量能力未完整打通

模板接口接收 `variables`，但发送时只按接收者名称做渲染。

结论：

- 模板能力当前只实现了基础版
- 还没有形成真正的自定义变量渲染链路

## 13. 推荐后续优化方向

### 13.1 让调度真正按任务状态和发送时间运行

建议定时任务取数时增加：

- 仅处理 `mass_task.send_status in (0, 1)` 的任务
- 仅处理 `send_time is null or send_time <= now()`
- 排除已取消任务

### 13.2 避免重复发送

建议：

- 手动触发时仅处理 `is_sent = 0` 的明细
- 或增加幂等控制

### 13.3 按 `msgType` 分流到不同发送接口

建议新增：

- 文本发送
- 图片发送
- 文件发送
- 音频发送
- 视频发送

并将 `mass_task` 中的素材字段真正用于群发发送。

### 13.4 完整支持模板变量

建议：

- 创建任务时保存模板变量快照
- 发送时基于该变量快照和接收方信息共同渲染

### 13.5 明确失败状态

建议将任务状态扩展为：

- `0` 待发送
- `1` 发送中
- `2` 已完成
- `3` 已取消
- `4` 执行失败

避免取消和异常共用同一状态。

## 14. 总结

当前群发任务功能已经具备以下核心能力：

- 支持圈人和圈群
- 支持创建任务主记录与发送明细
- 支持模板建任务
- 支持手动触发和定时轮询
- 支持逐条发送结果回写与任务统计

但当前实现本质上仍是“文本群发任务第一版”，在以下方面仍需继续完善：

- 真正的定时发送控制
- 取消任务生效范围
- 重复触发幂等性
- 多媒体群发
- 完整模板变量渲染

如果后续要继续扩展群发任务功能，建议优先处理调度条件、任务状态语义和多消息类型发送分流。

- `4`：视频消息

- `6`：小程序消息

以上定义已同步到以下位置：

- `MassTask.msgType`
- `MessageTemplate.msgType`
- `doc/sql/mass_message_tables.sql`

### 15.2 为前端提供的消息类型配置接口

为避免前端把“每种消息类型的必填项、可选项、是否支持存储、是否支持执行”写死在页面里，API 项目新增了两类辅助接口：

- `GET /api/v1/mass/message-type-specs`
  返回每种 `msgType` 的字段规范，包括：
  `requiredFields`、`optionalFields`、`storageSupported`、`executionSupported`、`notes`

- `POST /api/v1/mass/task/validate`
  接收富结构请求体，提前校验该次群发任务参数是否合法，便于前端在真正提交前先做表单拦截

前端建议接入方式：

1. 页面初始化时先调用 `GET /api/v1/mass/message-type-specs`
2. 根据返回的字段配置动态渲染表单和必填规则
3. 点击提交前先调用 `POST /api/v1/mass/task/validate`
4. 校验通过后，再调用正式的创建接口

### 15.3 新增的富结构请求模型

本次新增了一组用于参数规范化和前端联动的请求模型：

- `MassTaskCreateRequest`
- `MassTaskPayload`
- `MassTaskMediaPayload`
- `MassTaskLinkPayload`
- `MassTaskAppPayload`

结构约定如下：

- 文本消息使用 `payload.content`
- 图片 / 文件 / 语音 / 视频 使用 `payload.media`
- 链接卡片使用 `payload.link`
- 小程序消息使用 `payload.app`

其设计目标是：

- 让前端可以按消息类型统一组织数据
- 让后端在一个入口做集中校验
- 为后续把群发任务扩展到图片、语音、链接卡片、小程序等类型预留结构

### 15.4 各消息类型当前必填规则

后端校验服务 `MassTaskRequestValidator` 当前规则如下。

#### 15.4.1 文本消息

- 当 `templateId` 为空时，`payload.content` 必填

对应下游发送接口：

- `/wxwork/SendTextMsg`

#### 15.4.2 图片 / 文件 / 语音 / 视频

- `payload.media` 必填
- `payload.media.url` 与 `payload.media.base64` 二选一，至少传一个

补充说明：

- `payload.media.filename` 建议传，尤其是 base64 场景
- `payload.media.contentType` 建议传，便于识别媒体类型
- 语音场景下 `payload.media.voiceTime` 当前作为可选项，未传时可由后端后续按媒体内容推断

与下游接口的关系：

- 图片消息最终对应 `/wxwork/SendCDNImgMsg`
- 语音消息最终对应 `/wxwork/SendCDNVoiceMsg`
- 图片类 CDN 信息应参考 `CdnImageService`
- 文件 / 语音 / 视频 / 小程序封面类 CDN 信息应参考 `CdnFileService.uploadFile(String uuid, ReplyMediaItem filePayload)`

#### 15.4.3 链接卡片消息

- `payload.link.url` 必填
- `payload.link.title` 必填
- `payload.link.content` 必填
- `payload.link.imgurl` 当前为可选

对应下游发送接口：

- `/wxwork/SendLinkMsg`

#### 15.4.4 小程序消息

- `payload.app.desc` 必填
- `payload.app.appName` 必填
- `payload.app.title` 必填
- `payload.app.pagepath` 必填
- `payload.app.username` 必填
- `payload.app.appid` 必填
- `payload.app.cover.url` 与 `payload.app.cover.base64` 二选一，至少传一个
- `payload.app.weappIconUrl` 当前为可选

对应下游发送接口：

- `/wxwork/SendAppMsg`

补充说明：

- 小程序封面图上传后需要拿到 `cdnkey`、`md5`、`aeskey`、`fileSize` 等公共签名字段
- 这些字段应复用现有 CDN 上传服务生成，不建议前端自行拼装

### 15.5 旧版创建接口的能力边界

当前旧版创建接口仍然是：

- `POST /api/v1/mass/task`

该接口直接接收 `MassTask`，因此只能安全处理 `mass_task` 表中已经有独立字段承载的内容。

当前后端对旧接口的限制为：

- 文本消息：支持
- 图片消息：仅支持校验 `imageMediaId`
- 文件消息：仅支持校验 `fileMediaId`
- 语音消息：仅支持校验 `audioMediaId`
- 视频消息：仅支持校验 `videoMediaId`
- 链接卡片消息：不支持，调用时会直接返回校验失败
- 小程序消息：不支持，调用时会直接返回校验失败

也就是说：

- 新增的 `validate` / `specs` 接口已经能告诉前端“应该怎么传”
- 但旧版正式创建接口本身，还没有完成链接卡片、小程序这两类富结构内容的持久化与执行能力

### 15.6 当前“存储能力 / 执行能力”现状

按消息类型划分，当前状态如下：

- 文本消息
  - 存储能力：支持
  - 执行能力：支持

- 图片 / 文件 / 语音 / 视频
  - 存储能力：部分支持
  - 说明：当前 `mass_task` 表已有媒体 ID 类字段，但“基于 url/base64 自动上传并入库”的完整链路还未在群发任务主流程里全部打通
  - 执行能力：群发任务主流程尚未按 `msgType` 完整路由到对应下游发送接口

- 链接卡片
  - 存储能力：暂不支持
  - 说明：`mass_task` 表暂无链接卡片专属字段或统一 JSON 载荷字段
  - 执行能力：暂不支持

- 小程序
  - 存储能力：暂不支持
  - 说明：`mass_task` 表暂无小程序专属字段或统一 JSON 载荷字段
  - 执行能力：暂不支持

### 15.7 当前新增接口与正式发送接口的关系

本次新增接口主要解决的是“参数规范”和“前端联动”问题，不等于群发任务已经完整支持所有消息类型发送。

关系如下：

- `GET /api/v1/mass/message-type-specs`
  用于告诉前端：某个 `msgType` 需要哪些字段、哪些字段可选、当前是否支持入库和执行

- `POST /api/v1/mass/task/validate`
  用于告诉前端：这次提交的数据是否合法

- `POST /api/v1/mass/task`
  仍是当前正式落库入口，但受限于 `MassTask` 和 `mass_task` 表结构，只适合文本及现有媒体 ID 场景

- 下游企业微信发送接口
  包括 `/wxwork/SendTextMsg`、`/wxwork/SendCDNImgMsg`、`/wxwork/SendCDNVoiceMsg`、`/wxwork/SendLinkMsg`、`/wxwork/SendAppMsg`
  这些接口描述的是最终发送能力，不代表群发任务主流程已经全部接通

### 15.8 后续建议

如果下一步要把群发任务真正扩展到图片、语音、链接卡片、小程序，建议按下面顺序推进：

1. 为链接卡片、小程序补齐专属持久化字段，或新增统一 JSON payload 存储方案
2. 正式创建接口改为接收富结构请求模型，而不是只接收 `MassTask`
3. 在群发任务执行阶段，按 `msgType` 路由到对应下游发送接口
4. 对图片、语音、小程序封面等媒体，统一在发送前走 CDN 上传服务，获取 `cdnkey`、`aeskey`、`md5`、`fileSize` 等签名信息

## 16. 2026-04-13 推荐表结构补充

针对“完整支持图片、文件、语音、视频、链接卡片、小程序群发任务”的目标，`doc/sql/mass_message_tables.sql` 已同步调整为推荐完整版表结构。

本次推荐的核心思路不是继续为每一种消息类型单独无限加字段，而是：

- 保留 `content`、`image_media_id`、`file_media_id`、`audio_media_id`、`video_media_id` 这些历史字段，兼容旧逻辑
- 新增 `payload_json`，作为富结构消息内容的统一存储入口
- 新增 `payload_version`，便于后续升级 payload 结构
- 新增 `template_variables_json`，用于保存任务创建时的模板变量快照
- 新增 `fail_reason`，区分任务失败与取消等状态
- 在 `mass_task_detail` 中新增 `message_snapshot_json` 和 `downstream_response`，便于排查单个接收人的发送内容与下游响应

### 16.1 为什么推荐 `payload_json`

如果继续沿用“每种消息一个字段组”的方式，后续会很快遇到这些问题：

- 链接卡片需要 `url`、`title`、`content`、`imgurl`
- 小程序需要 `desc`、`appName`、`title`、`pagepath`、`username`、`appid`、`weappIconUrl`
- 小程序封面、图片、语音等媒体还要再带 CDN 上传后的签名信息

这类字段继续平铺到主表里，后续维护和扩展成本会越来越高。因此推荐把“消息业务载荷”统一收口到 `payload_json`。

### 16.2 推荐后的表职责

- `mass_task`
  负责保存任务级别信息、消息类型、统一 payload、模板快照、整体状态

- `mass_task_detail`
  负责保存接收人维度的发送结果、每个接收人的消息快照、下游响应

- `message_template`
  负责保存模板定义；文本模板继续可用 `template_content`，富消息模板统一进入 `payload_json`

### 16.3 需要注意的事实

这里同步的是“推荐完整表设计”，不是说当前正式代码已经全部按这套结构落地。

当前代码现状仍然是：

- 参数规范与前端联动接口已经补齐
- 文本群发链路可用
- 多媒体 / 链接卡片 / 小程序在群发任务主链路里仍待进一步接通

也就是说，`doc/sql/mass_message_tables.sql` 现在表达的是下一阶段应采用的目标表结构。
